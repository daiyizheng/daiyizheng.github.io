Rasa文档指南

安装

  • 快速安装

您可以使用pip安装Rasa Open Source（需要Python 3.6或3.7）。

pip3 install rasa
  • 从源代码构建
curl -sSL https://raw.githubusercontent.com/python-poetry/poetry/master/get-poetry.py | python
git clone https://github.com/RasaHQ/rasa.git
cd rasa
poetry install

  • spaCy的依赖关系

    有关spaCy的更多信息，请查看spaCy文档。

pip install rasa[spacy]
python -m spacy download en_core_web_md
python -m spacy link en_core_web_md en

  • MITIE的依赖项

    然后下载 MITIE模型。您需要的文件是total_word_feature_extractor.dat。将此保存在任何地方。如果要使用MITIE，则需要告诉它在哪里可以找到此文件（在本示例中，该data文件已保存在项目目录的 文件夹中）。

pip install git+https://github.com/mit-nlp/MITIE.git
pip install rasa[mitie]

教程：Rasa基础

基本步骤

  1. 创建一个新项目

  2. 查看您的NLU培训数据

在本教程中，您将构建一个简单友好的助手，该助手将询问您的工作状况，并向您发送有趣的图片以在您难过时为您加油。

1.创建一个新项目

第一步是创建一个新的Rasa项目。为此，请运行：

rasa init --no-prompt

该命令创建Rasa项目所需的所有文件，并在一些示例数据上训练一个简单的bot。如果您不做任何标记，系统将询问您一些有关如何设置项目的问题。rasa init --no-prompt

这将创建以下文件：

__init__.py 一个空文件，可帮助python查找您的操作
actions.py 您的自定义操作的代码
config.yml ‘*’ 您的NLU和Core模型的配置
credentials.yml 连接其他服务的详细信息
data/nlu.md ‘*’ 您的NLU训练数据
data/stories.md ‘*’ 你的故事
domain.yml ‘*’ 您助手的域名
endpoints.yml 连接到fb Messenger等频道的详细信息
models/<timestamp>.tar.gz 您的初始模型

最重要的文件标有“ *”。您将在本教程中了解所有这些内容。

2.查看您的NLU培训数据

Rasa助手的第一部分是NLU模型。NLU代表自然语言理解，这意味着将用户消息转换为结构化数据。要使用Rasa做到这一点，您需要提供训练示例，这些示例说明Rasa应该如何理解用户消息，然后通过显示这些示例来训练模型。

运行下面的代码单元以查看该命令创建的NLU训练数据：rasa init

## intent:greet

- hello
- hi
- good morning
- good evening
- hey there

## intent:goodbye
- bye
- goodbye
- see you around
- see you later

## intent:affirm

以开头的行##定义your的名称intents，它们是具有相同含义的消息组。当您的用户向您的助手发送看不见的新消息时，Rasa的工作将是预测正确的意图。您可以在训练数据格式中找到该数据格式的所有详细信息。

3.定义你的模型配置

配置文件定义了模型将使用的NLU和Core组件。在此示例中，您的NLU模型将使用 supervised_embeddings管道。您可以在此处了解不同的NLU管道 。

让我们看一下您的模型配置文件。

# Configuration for Rasa NLU.
# https://rasa.com/docs/rasa/nlu/components/
language: en
pipeline:
  - name: WhitespaceTokenizer
  - name: RegexFeaturizer
  - name: LexicalSyntacticFeaturizer
  - name: CountVectorsFeaturizer
  - name: CountVectorsFeaturizer
    analyzer: "char_wb"
    min_ngram: 1
    max_ngram: 4
  - name: DIETClassifier
    epochs: 10
  - name: EntitySynonymMapper
  - name: ResponseSelector
    epochs: 10

# Configuration for Rasa Core.
# https://rasa.com/docs/rasa/core/policies/
policies:
  - name: MemoizationPolicy
  - name: TEDPolicy
    max_history: 5
    epochs: 10
  - name: MappingPolicy

在language和pipeline主键指定的NLU模型应该如何构建。该policies键定义策略的核心车型将采用。

4.编写第一个故事

在此阶段，您将教您的助手如何回复您的消息。这称为对话管理，由您的Core模型处理。

核心模型以训练“故事”的形式从真实的对话数据中学习。故事是用户和助手之间的真实对话。具有意图和实体的行反映了用户的输入，而动作名称则显示了助手应该做出的响应。

下面是一个简单对话的示例。用户打个招呼，助手打个招呼。这就是一个故事：

## story1
* greet
   - utter_greet

您可以在“故事”中查看完整的详细信息。

开头的行是-助手执行的操作。在本教程中，我们所有的动作都是发送回用户的消息，例如utter_greet，但是一般而言，动作可以执行任何操作，包括调用API和与外界互动。

运行以下命令以查看文件中的示例故事data/stories.md：

## happy path
* greet
  - utter_greet
* mood_great
  - utter_happy

## sad path 1
* greet
  - utter_greet
* mood_unhappy
  - utter_cheer_up
  - utter_did_that_help
* affirm
  - utter_happy

## sad path 2
* greet
  - utter_greet
* mood_unhappy
  - utter_cheer_up
  - utter_did_that_help
* deny
  - utter_goodbye

## say goodbye
* goodbye
  - utter_goodbye

## bot challenge
* bot_challenge
  - utter_iamabot

5.定义一个域

我们需要做的下一件事是定义一个Domain。该域定义了您的助手所生活的世界：应该期望得到什么用户输入，应该能够预测什么操作，如何响应以及要存储什么信息。我们助手的域保存在一个名为的文件中domain.yml：

intents:
  - greet
  - goodbye
  - affirm
  - deny
  - mood_great
  - mood_unhappy
  - bot_challenge

responses:
  utter_greet:
  - text: "Hey! How are you?"

  utter_cheer_up:
  - text: "Here is something to cheer you up:"
    image: "https://i.imgur.com/nGF1K8f.jpg"

  utter_did_that_help:
  - text: "Did that help you?"

  utter_happy:
  - text: "Great, carry on!"

  utter_goodbye:
  - text: "Bye"

  utter_iamabot:
  - text: "I am a bot, powered by Rasa."

session_config:
  session_expiration_time: 60
  carry_over_slots_to_new_session: true

那么不同的部分是什么意思呢？

intents 您希望用户说的话
actions 您的助手可以说的事情
responses 您的助手可以说的话的回应字符串

这如何搭配在一起？ Rasa Core的工作是选择正确的动作，以便在对话的每个步骤中执行。在这种情况下，我们的操作只会向用户发送一条消息。这些简单的发声动作是以actions开头的域utter_。助手将根据此responses部分的响应以一条消息进行响应。请参阅自定义操作 ，以构建除发送消息以外还可以做的事情。

6.训练模型

每当我们添加新的NLU或Core数据，或更新域或配置时，我们都需要根据示例故事和NLU数据重新训练神经网络。为此，请运行以下命令。该命令将调用Rasa Core和NLU训练函数，并将训练后的模型存储到models/目录中。如果不同模型零件的数据或配置有所更改，该命令将仅自动重新训练它们。

rasa train

该命令将同时查找NLU和Core数据，并训练组合模型。rasa train

7.测试您的助手

训练模型后，您始终希望检查助手的行为是否仍然符合您的期望。在Rasa开放源代码中，您可以使用tests/目录中定义的端到端测试来运行测试对话，以确保NLU和Core都能做出正确的预测。

rasa test

请参阅测试您的助手以了解有关在改进模型时如何评估模型的更多信息。

8.与您的助手交谈

恭喜你！🚀您刚刚构建了一个完全由机器学习驱动的助手。

下一步就是尝试一下！如果您在本地计算机上遵循本教程，请运行以下命令开始与助手交谈：

rasa shell

构建一个简单的FAQ助手

FAQ助手是最简单的助手，也是入门的好地方。这些助手允许用户提出一个简单的问题并得到答复。我们将使用专门为此类助手设计的Rasa功能来构建基本的FAQ助手。

在本节中，我们将介绍以下主题：

  • 通过MemoizationPolicy响应简单的意图
  • 使用ResponseSelector处理常见问题

为了准备本教程，我们将创建一个新目录并启动一个新的Rasa项目。

mkdir rasa-assistant
rasa init

让我们从bot中删除默认内容，以便nlu.md，stories.md 和domain.yml文件为空。

Memoization Policy

MemoizationPolicy最多可以记住训练故事中的示例max_history 。“turns”次数包括用户发送的消息以及助手执行的操作。出于简单，无上下文的FAQ机器人的目的，我们只需要注意用户发送的最后一条消息，因此我们将其设置为1。

您可以按照以下步骤编辑config.yml文件来完成此操作（现在可以删除TEDPolicy）：

policies:
- name: MemoizationPolicy
  max_history: 1
- name: MappingPolicy

注意

MappingPolicy之所以存在，是因为它处理了/restart意图的逻辑，这使您可以清除对话历史记录并重新开始。

现在，我们已经定义了策略，我们可以为文件添加一些故事goodbye，thank并向文件添加greet 意图stories.md：

## thank

responses:
  utter_noworries:
    - text: No worries!
  utter_greet:

响应选择器

该ResponseSelector NLU组件的设计，使其更容易处理对话元素，如小对话和常见问题的信息以简单的方式。通过使用ResponseSelector，您只需要一个故事即可处理所有FAQ，而无需在每次想要增加机器人的范围时添加新故事。

人们常常问Rasa周边Rasa产品不同的问题，所以让我们开始有三个意图：ask_channels，ask_languages，和ask_rasax。我们将从Sara训练数据中复制一些NLU数据 到我们的nlu.md。这些意图必须带有faq/前缀，这一点很重要，因此ResponseSelector会将它们识别为常见问题意图：

## intent: faq/ask_channels
- What channels of communication does rasa support?
- what channels do you support?
- what chat channels does rasa uses
- channels supported by Rasa
- which messaging channels does rasa support?

## intent: faq/ask_languages
- what language does rasa support?
- which language do you support?
- which languages supports rasa
- can I use rasa also for another laguage?
- languages supported

## intent: faq/ask_rasax
- I want information about rasa x
- i want to learn more about Rasa X
- what is rasa x?
- Can you tell me about rasa x?
- Tell me about rasa x
- tell me what is rasa x

接下来，我们需要responses.md在data/目录中的新文件中定义与这些FAQ相关的响应：

* faq/ask_channels
  - We have a comprehensive list of [supported connectors](https://rasa.com/docs/core/connectors/), but if
    you don't see the one you're looking for, you can always create a custom connector by following
    [this guide](https://rasa.com/docs/rasa/user-guide/connectors/custom-connectors/).

## ask languages
* faq/ask_languages
  - You can use Rasa to build assistants in any language you want!

## ask rasa x
* faq/ask_rasax
 - Rasa X is a tool to learn from real conversations and improve your assistant. Read more [here](https://rasa.com/docs/rasa-x/)

ResponseSelector应该已经在我们的NLU管道的末尾config.yml：

language: en
pipeline:
  - name: WhitespaceTokenizer
  - name: RegexFeaturizer
  - name: LexicalSyntacticFeaturizer
  - name: CountVectorsFeaturizer
  - name: CountVectorsFeaturizer
    analyzer: "char_wb"
    min_ngram: 1
    max_ngram: 4
  - name: DIETClassifier
    epochs: 100
  - name: EntitySynonymMapper
  - name: ResponseSelector
    epochs: 100

现在我们已经定义了NLU端，我们需要让Core意识到这些变化。打开domain.yml文件并添加faq意图：

intents:
  - greet
  - bye
  - thank
  - faq

我们还需要添加一个检索动作，该动作负责将ResponseSelector预测的响应发送回用户到动作列表。这些操作始终必须以respond_前缀开头：

actions:
  - respond_faq

接下来，我们将编写一个story，以便Core知道要预测的动作：

## Some question from FAQ
* faq

如我们前面所述，此预测由MemoizationPolicy处理。

rasa shell

构建一个上下文助手

无论您是刚刚创建FAQ机器人还是从头开始，下一步都是扩展您的bot来处理上下文对话。

在本教程中，我们将涵盖各种主题：

  • 处理业务逻辑
  • 处理意外的用户输入
  • 优雅失败
  • 更复杂的上下文对话

在开始本部分之前，请确保您已从“构建简单的FAQ助手”部分获取了所有数据。您将需要对配置文件进行一些调整，因为我们现在需要注意上下文：

policies:
- name: MemoizationPolicy
- name: MappingPolicy

我们删除了配置。默认值为，这意味着Core进行预测时会注意过去5圈（请参阅最大历史记录的说明）。max_history: 15

处理业务逻辑

许多对话助手的用户目标都是在能够为用户做某事之前先从用户那里收集一堆信息。这称为插槽填充。例如，在银行业中，您可能有一个转移资金的用户目标，在这里您需要收集有关从哪个帐户转账，向谁转账以及转账金额的信息。可以而且应该以基于规则的方式处理这种行为，因为很清楚应该如何收集此信息。

对于这种类型的用例，我们可以使用Forms和FormPolicy。该FormPolicy 的工作原理是，直到所有的信息从用户收集预测形式下一个动作。

作为示例，我们将构建Sara的SalesForm。用户想要联系我们的销售团队，为此，我们需要收集以下信息：

  • 他们的工作
  • 他们的机器人用例
  • 他们的名字
  • 他们的电子邮件
  • 他们的预算
  • 他们的公司

我们将从SalesForm在名为的文件中将其定义为新类开始actions.py。我们需要定义的第一个方法是名称，就像在常规Action中一样，它返回将在我们的story中使用的名称：

from rasa_sdk.forms import FormAction

class SalesForm(FormAction):
    """Collects sales information and adds it to the spreadsheet"""

    def name(self):
        return "sales_form"

接下来，我们必须定义required_slots一种方法，该方法指定要询问的信息，即要填充的插槽。

@staticmethod
def required_slots(tracker):
    return [
        "job_function",
        "use_case",
        "budget",
        "person_name",
        "company",
        "business_email",
        ]

注意：您可以将所需的插槽功能自定义为非静态。例如，如果job_function是开发人员，则可以添加required_slot与Rasa有关的用户体验级别

完成此操作后，您需要指定机器人应如何请求此信息。这是通过utter_ask_{slotname}在domain.yml文件中指定响应来完成的。对于上述内容，我们需要指定以下内容

utter_ask_business_email:
  - text: What's your business email?
utter_ask_company:
  - text: What company do you work for?
utter_ask_budget:
  - text: "What's your annual budget for conversational AI? 💸"
utter_ask_job_function:
  - text: "What's your job? 🕴"
utter_ask_person_name:
  - text: What's your name?
utter_ask_use_case:
  - text: What's your use case?

我们还需要在domain.yml文件中定义所有这些插槽：

def submit(

        tracker: Tracker,
        domain: Dict[Text, Any],

    dispatcher.utter_message("Thanks for getting in touch, we’ll contact you soon")
    return []

另一个示例是，您可能想填充名称相同的实体以外的东西的插槽。例如，对于我们表单中的“用例”情况，我们希望用户键入完整的句子，而不是您可能必须提取为实体的东西。在这种情况下，我们可以利用该slot_mappings方法，您可以在其中描述应从中提取实体的内容。在这里，我们可以使用该from_text方法提取用户的整个消息：

def slot_mappings(self) -> Dict[Text, Union[Dict, List[Dict[Text, Any]]]]:
    """A dictionary to map required slots to
    - an extracted entity
    - intent: value pairs
    - a whole message
    or a list of them, where a first match will be picked"""
    return &#123;"use_case": self.from_text(intent="inform")&#125;

现在，当请求用例插槽时，我们的机器人将提取完整的用户消息，并且我们不需要使用use_case之前定义的实体。

可以定制表单中的所有方法，以处理业务逻辑中的不同分支。在此处阅读有关此内容的更多信息。但是，您应确保不要处理表单内所有不愉快的路径。应该通过编写常规故事来处理这些问题，以便您的模型可以学习这种行为。

这是我们为使用表单操作处理业务逻辑而修改的文件的最小清单：

  • actions.py：定义形式作用，包括required_slots，slot_mappings和submit方法

  • data/nlu.md：

    • 添加示例以激活表单
    • 添加示例inform以填写表单

  • domain.yml：

    • 添加表格所需的所有插槽
    • 添加utter_ask_{slot}所有必需插槽的响应
    • 将表单动作添加到该forms部分
    • 从您的NLU培训数据中添加所有意图和实体

  • data/stories.md：为表单添加一个故事

  • config.yml：

    • 将新增FormPolicy至您的政策
    • 将实体提取器添加到管道

  • endpoints.yml：定义 action_endpoint

处理意想不到的用户输入

所有预期的用户输入都应通过我们上面定义的形式处理，即，如果用户提供了机器人所要求的信息。但是，在实际情况下，用户的行为通常会有所不同。在本节中，我们将介绍各种形式的“感叹词”以及如何在Rasa中处理它们。

处理这些类型的用户输入的决定应始终来自查看真实的对话。您应该首先构建助手的一部分，与真实用户（无论是最终用户还是同事）进行测试，然后添加缺失的内容。您不应该尝试实施您认为可能发生的所有可能的极端情况，因为最终用户可能永远不会以这种方式行事。 Rasa X 是可以帮助您查看对话并做出此类决定的工具。

通用感叹词

如果您有通用感叹词，无论上下文如何，都应始终具有相同的单个响应，则可以使用映射策略来处理这些感叹词。它将始终为意图预测相同的动作，并且与遗忘机制结合使用时，您也无需编写任何故事。

例如，假设您看到用户与您的助手进行如下对话，他们在对话中间写了问候-可能是因为他们离开了几分钟：

问候意图是一个很好的例子，我们将始终给出相同的响应，但是我们不希望这种意图影响对话历史。为此，响应必须是返回UserUtteranceReverted()事件以从对话历史记录中删除交互的动作。

首先，打开domain.yml文件并修改问候意图，然后actions在文件中添加一个新块，然后，action_greet如下所示添加：

intents:
  - greet: {triggers: action_greet}
  - bye
  - thank
  - faq
  - contact_sales
  - inform

actions:
  - action_greet

如果有，请使用“打招呼”意图删除所有故事。

接下来，我们需要定义action_greet。将以下操作添加到actions.py文件中：

from rasa_sdk import Action
from rasa_sdk.events import UserUtteranceReverted

class ActionGreetUser(Action):
"""Revertible mapped action for utter_greet"""

def name(self):
    return "action_greet"

def run(self, dispatcher, tracker, domain):
    dispatcher.utter_template("utter_greet", tracker)
    return [UserUtteranceReverted()]

然后，我们可以重新训练模型，并尝试添加以下内容：

rasa shell

为了解决这个问题，我们需要使requested_slot特征化，并为其分配分类类型：

正常失败

即使您完美地设计了机器人，用户也不可避免地会向您的助手说出您意料之外的事情。在这些情况下，您的助手将失败，因此请确保其正常运行很重要。

后备政策

最常见的故障之一是NLU置信度低，使用TwoStageFallbackPolicy可以很好地处理它。您可以通过将以下内容添加到配置文件中来启用它，

policies:
  - name: TwoStageFallbackPolicy
    nlu_threshold: 0.8

并将out_of_scope意图添加到domain.yml文件中：

intents:
- out_of_scope

当nlu置信度降至定义的阈值以下时，机器人将提示用户重新表达其消息。如果僵尸程序无法三遍获取其消息，则将采取最终行动，例如，僵尸程序可以移交给人类。

要尝试此操作，请重新训练您的模型，然后向您的机器人发送“向我订购披萨”之类的消息：

rasa train
rasa shell

您还可以通过多种方式自定义此策略。在我们的演示机器人Sara中，我们对其进行了自定义，以在一定的置信度范围内向用户建议意图，以使用户更轻松地向机器人提供所需的信息。

这是通过自定义动作来完成的，ActionDefaultAskAffirmation如Sara rasa-demo动作服务器中所示。 我们定义了一些意图映射，以使用户对意图的理解更加直观。

输出范围意图

优良作法是也要处理您知道用户可能会问的问题，但您尚未针对这些问题实现用户目标。

您可以定义一个out_of_scope意图来处理超出范围的通用请求，例如“我饿了”，并让漫游器响应默认消息，例如“抱歉，我无法处理该请求”：

* out_of_scope
  utter_out_of_scope

我们还需要为out_of_scope意图添加NLU数据：

## intent:out_of_scope
- I want to order food
- What is 2 + 2?
- Who’s the US President?
- I need a job

最后，我们将对domain.yml文件添加响应：

responses:
  utter_out_of_scope:
  - text: Sorry, I can’t handle that request.

现在，我们可以重新训练并测试此添加项

rasa train
rasa shell

再往前走，如果您发现用户要求某些事情，将来又希望成为用户目标，则可以将它们作为单独的意图进行处理，以使用户知道您已经理解了他们的信息，但是还没有解决方案。例如，假设用户问“我想在Rasa申请工作”，然后我们可以回答“我了解您正在寻找工作，但我恐怕还无法处理该技能。”

* ask_job
  utter_job_not_handled

注意

这是我们为帮助助手正常退出而修改的文件的最小清单：

  • data/nlu.md：

    • 添加out_of_scope意图和要单独处理的任何特定范围外意图的训练数据

  • data/stories.md：

    • 为任何超出范围的意图添加故事

  • domain.yml：

    • 添加out_of_scope意图和任何超出范围的意图
    • utter_out_of_scope为任何超出范围的意图添加一个或多个响应

  • actions.py：

    • 自定义ActionDefaultAskAffirmation建议用户的意图，供用户选择

  • config.yml：

    • 将TwoStageFallbackPolicy添加到该policies部分

更复杂的上下文对话

并非您定义的每个用户目标都将属于业务逻辑类别。对于其他情况，您将需要使用故事和上下文来帮助用户实现其目标。

如果我们以Sara的“入门”技能为例，我们想根据他们是否曾经构建过AI助手并且正在从其他工具迁移而来，向他们提供不同的信息。这可以通过故事很简单地完成和最大历史的概念。

用ML概括

除了上文所述的基于规则的策略外，Core还提供了一些可以使用的ML策略。这些作为策略配置中的附加层进入，只有在用户遵循您未曾预料到的路径时才跳入。重要的是要了解使用这些策略并不意味着放开对助手的控制权。如果基于规则的策略能够做出预测，则该预测将始终具有更高的优先级（在此处了解更多信息）并预测下一个操作。基于ML的策略为您的助手提供了不失败的机会，而如果不使用它们，您的助手肯定会失败，就像在基于状态机的对话系统中一样。

这些类型的意外用户行为是我们的TEDPolicy非常擅长处理的事情。它可以学习在用户试图完成的主要用户目标中的一些插词之后使用户回到正轨。例如，在下面的对话中（摘自Rasa X上的对话）：

## Story from conversation with a2baab6c83054bfaa8d598459c659d2a on November 28th 2019
* greet
  - action_greet_user
  - slot&#123;"shown_privacy":true&#125;
* ask_whoisit
  - action_chitchat
* ask_whatspossible
  - action_chitchat
* telljoke
  - action_chitchat
* how_to_get_started&#123;"product":"x"&#125;
  - slot&#123;"product":"x"&#125;
  - utter_explain_x
  - utter_also_explain_nlucore
* affirm
  - utter_explain_nlu
  - utter_explain_core
  - utter_direct_to_step2

在这里，我们可以看到用户首先完成了一些闲聊任务，然后最终询问他们如何开始使用RasaX。TEDPolicy正确地预测了应该向用户解释Rasa X，然后也将其沿用下来路径，而不先问所有符合条件的问题。

由于在这种情况下机器学习策略的推广效果很好，因此有必要将此故事添加到您的训练数据中，以不断改进您的机器人并帮助机器学习进一步推广。Rasa X是一款可以帮助您改善bot并使它与上下文相关的工具。

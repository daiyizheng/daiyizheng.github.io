使用Rasa的HTTP

注意

以下说明与配置模型如何在Docker容器中运行或在本地测试HTTP API有关。如果要向用户部署助手，请参阅部署Rasa助手。

您可以运行一个简单的HTTP服务器，该服务器使用训练有素的Rasa模型处理以下请求：

rasa run -m models --enable-api --log-file out.log

该API公开的所有端点都记录在HTTP API中。

不同的参数是：

  • -m：包含Rasa模型的文件夹的路径，
  • --enable-api：启用此附加API，并
  • --log-file：日志文件的路径。

Rasa可以通过三种不同的方式加载模型：

  1. 从服务器获取模型（请参阅从服务器获取模型），或者
  2. 从远程存储中获取模型（请参阅Cloud Storage）。
  3. 通过-m本地存储系统加载指定的模型，

Rasa尝试以上述顺序加载模型，即，如果未配置任何模型服务器和远程存储，它只会尝试从本地存储系统加载模型。

警告

确保通过限制对服务器的访问（例如使用防火墙）或启用身份验证方法来确保服务器的安全：安全注意事项。

注意

如果您正在使用自定义动作，请确保您的动作服务器正在运行（请参阅启动动作服务器）。如果您的操作在其他计算机上运行，或者您没有使用Rasa SDK，请确保更新endpoints.yml文件。

注意

如果使用仅NLU模型启动服务器，则无法调用所有可用端点。请注意，某些端点将返回409状态代码，因为需要训练有素的Core模型来处理请求。

注意

默认情况下，HTTP服务器作为单个进程运行。您可以使用SANIC_WORKERS环境变量更改工作进程的数量。建议您将工作程序数设置为可用的CPU内核数（ 有关更多详细信息，请查看 Sanic文档）。这只能与结合使用 RedisLockStore（请参阅锁存储）

从服务器获取模型

您可以将HTTP服务器配置为从另一个URL获取模型：

rasa run --enable-api --log-file out.log --endpoints my_endpoints.yml

在端点配置（my_endpoints.yml）中指定模型服务器，在该配置中您可以指定服务器URL Rasa定期查询压缩的Rasa模型：

models:
  url: http://my-server.com/models/default@latest
  wait_time_between_pulls: 10   # [optional](default: 100)

注意

如果只想从服务器提取一次模型，请设置 wait_time_between_pulls为None。

注意

您的模型服务器必须提供压缩的Rasa模型，并且 标头之一。仅当此模型哈希值已更改时，Rasa才会下载新模型。{"ETag": <model_hash_string>}

Rasa使用If-None-Match 包含当前模型哈希的标头将请求发送到模型服务器。如果您的模型服务器可以为模型提供与您发送的哈希不同的哈希，则它应以zip文件的形式发送，并带有ETag包含新哈希的标头。如果不是，则Rasa期望带有204或304状态代码的空响应。

Rasa可能向模型服务器发出的示例请求如下所示：

 curl --header "If-None-Match: d41d8cd98f00b204e9800998ecf8427e" http://my-server.com/models/default@latest

从远程存储获取模型

您还可以配置Rasa服务器以从远程存储中获取模型：

rasa run -m 20190506-100418.tar.gz --enable-api --log-file out.log --remote-storage aws

该模型将被下载并存储在本地存储系统上的临时目录中。有关更多信息，请参阅Cloud Storage。

配置SSL /

默认情况下，Rasa服务器使用HTTP进行通信。为了确保与SSL的通信安全，您需要提供有效的证书和相应的私钥文件。

您可以在命令中指定这些文件：rasa run

rasa run --ssl-certificate myssl.crt --ssl-keyfile myssl.key

如果在创建过程中使用密码对密钥文件进行了加密，则需要将此密码添加到命令中：

rasa run --ssl-certificate myssl.crt --ssl-keyfile myssl.key --ssl-password mypassword

安全注意事项

我们建议不要将Rasa Server暴露给外界，而应通过专用连接（例如docker容器之间）从后端连接到Rasa Server。

但是，内置了两种身份验证方法：

基于令牌的身份验证：

启动服务器时使用传递令牌：--auth-token thisismysecret

rasa run \
    -m models \
    --enable-api \
    --log-file out.log \
    --auth-token thisismysecret

在我们的示例中thisismysecret，您的请求应将令牌作为参数传递：

 curl -XGET localhost:5005/conversations/default/tracker?token=thisismysecret

基于JWT的身份验证：

使用启用基于JWT的身份验证。发送到服务器的请求需要在使用此机密和算法签名的标头中包含有效的JWT令牌。--jwt-secret thisismysecret Authorization HS256

用户必须具有username和role属性。如果role为admin，则所有端点均可访问。如果role为user，sender_id则只有sender_id与用户的匹配时，才可以访问带有参数的端点username。

rasa run \
    -m models \
    --enable-api \
    --log-file out.log \
    --jwt-secret thisismysecret

您的请求应设置正确的JWT标头：

"Authorization": "Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ"
                 "zdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIi"
                 "wiaWF0IjoxNTE2MjM5MDIyfQ.qdrr2_a7Sd80gmCWjnDomO"
                 "Gl8eZFVfKXA6jhncgRn-I"

端点配置

要将Rasa连接到其他终结点，可以在YAML文件中指定终结点配置。然后使用flag运行Rasa 。--endpoints <path to endpoint configuration.yml>

例如：

rasa run \
    --m  \
    --endpoints .yml

注意

您可以通过在中指定配置文件来使用环境变量。然后将这些占位符替换为环境变量的值。${name of environment variable}

端到端测试

Rasa Open Source使您可以通过运行测试对话并确保NLU和Core做出正确的预测来端对端地测试对话。

为此，您需要一些端到端格式的故事，其中包括NLU输出和原始文本。这里有些例子：

基本
## A basic end-to-end test
* greet: hello
   - utter_ask_howcanhelp
* inform: show me [chinese](cuisine) restaurants
   - utter_ask_location
* inform: in [Paris](location)
   - utter_ask_price


   自定义动作
   ## End-to-End tests where a custom action appends events
* greet: hi
    - my_custom_action
    <!-- The following events are emitted by `my_custom_action` -->
    - slot&#123;"my_slot": "value added by custom action"&#125;
    - utter_ask_age
* thankyou: thanks
    - utter_noworries

Forms Happy Path
    ## Testing a conversation with a form
* greet: hi
    - utter_greet
* request_restaurant: im looking for a restaurant
    - restaurant_form
    - form&#123;"name": "restaurant_form"&#125;
* inform: [afghan](cuisine) food
    - form: restaurant_form
    - form&#123;"name": null&#125;
    - utter_slots_values
* thankyou: thanks
    - utter_noworries


    Forms Unhappy Path
    ## Testing a conversation with a form and unexpected user input
* greet: hi
    - utter_greet
* request_restaurant: im looking for a restaurant
    - restaurant_form
    - form&#123;"name": "restaurant_form"&#125;
<!-- The user sends a message which should not be handled by the form. -->
* chitchat: can you share your boss with me?
    - utter_chitchat
    - restaurant_form
    - form&#123;"name": null&#125;
    - utter_slots_values
* thankyou: thanks
    - utter_noworries

默认情况下，Rasa Open Source将会话测试保存到tests/conversation_tests.md。您可以通过运行以下命令来对您的助手进行测试：

rasa test

注意

自定义操作不会作为端到端测试的一部分执行。如果您的自定义操作将任何事件附加到跟踪器，则必须在端到端测试中反映出来（例如，通过将slot事件添加到端到端故事中）。

如果您有任何疑问或问题，请在我们论坛的专用测试部分与我们分享 ！

注意

请在确保您的模型文件models的组合core 和nlu型号。如果不包含NLU模型，则Core将使用default RegexInterpreter。

评估一个NLU模型

机器学习的标准技术是将一些数据分开作为测试集。您可以使用以下方法将NLU训练数据 分为训练集和测试集：

rasa data split nlu

如果完成了此操作，则可以使用以下命令查看NLU模型对测试用例的预测程度：

rasa test nlu -u train_test_split/test_data.md --model models/nlu-20180323-145833.tar.gz

如果您不想创建单独的测试集，则仍可以使用交叉验证来估计模型的概括程度。为此，添加标志--cross-validation：

rasa test nlu -u data/nlu.md --config config.yml --cross-validation

比较NLU管道

通过将多个管道配置（或包含它们的文件夹）传递到CLI，Rasa将在管道之间进行比较检查。

 rasa test nlu --config pretrained_embeddings_spacy.yml supervised_embeddings.yml
  --nlu data/nlu.md --runs 3 --percentages 0 25 50 70 90

上面示例中的命令将根据您的数据创建训练/测试拆分，然后使用训练集中排除的意图数据的0％，25％，50％，70％和90％多次训练每个管道。然后在测试集上评估模型，并记录每个排除百分比的f1得分。此过程运行了3次（即总共有3个测试集），然后使用f1得分的平均值和标准偏差绘制了图表。

f1得分图以及所有训练/测试集，训练有素的模型，分类和错误报告将保存到名为的文件夹中nlu_comparison_results。

意向分类

评估脚本将为您的模型生成报告，混淆矩阵和置信度直方图。

该报告记录每个意图和实体的精度，召回率和f1度量，并提供总体平均值。您可以使用--report参数将这些报告另存为JSON文件。

混淆矩阵向您显示哪些意图被误认为其他意图；任何被错误预测的样本都会被记录并保存到一个文件中errors.json，以便于调试。

脚本生成的直方图使您可以可视化所有预测的置信度分布，正确和不正确的预测量分别由蓝色和红色条显示。提高训练数据的质量会将蓝色直方图条移到该图的右侧，将红色直方图条移至该图的左侧。

警告

如果您的任何实体注释不正确，则评估可能会失败。一个常见的问题是实体无法在令牌内部停止或启动。例如，如果您有一个name类似的实体的示例，则仅在您的令牌生成器拆分为多个令牌时才有效。在这种情况下，空白标记器将不起作用。[Brian](name)'s house``Brian's

响应选择

评估脚本将为管道中的所有响应选择器模型生成组合报告。

该报告记录每个响应的精度，召回率和f1度量，并提供总体平均值。您可以使用--report参数将这些报告另存为JSON文件。

实体提取

，这CRFEntityExtractor是您使用自己的数据训练的唯一实体提取器，因此也是唯一要评估的实体提取器。如果您使用spaCy或小鸭经过预训练的实体提取器，则Rasa NLU将不会在评估中包括这些提取器。

Rasa NLU将报告受CRFEntityExtractor训练识别的每种实体类型的召回率，精度和f1度量 。

实体计分

为了评估实体提取，我们应用了一种基于标签的简单方法。我们不考虑BILOU标签，而仅考虑每个令牌基础上的实体类型标签。对于像“ Alexanderplatz附近”这样的位置实体，我们希望使用标签而不是基于BILOU的标签。对于评估，我们的方法更为宽容，因为它奖励部分提取，并且不会惩罚实体的分裂。例如，给定前述实体“ Alexanderplatz附近”和提取“ Alexanderplatz”的系统，我们的方法将奖励提取“ Alexanderplatz”并惩罚丢失的单词“ near”。但是，基于BILOU的方法会将其标记为完全失败，因为它希望将Alexanderplatz标记为实体（）中的最后一个标记，而不是单个标记实体（LOC LOC``B-LOC L-LOC``L-LOC``U-LOC）。还要注意，将“ near”和“ Alexanderplatz”分开提取将在我们的方法中获得满分，而在基于BILOU的方法中获得零分。

这是短语“near Alexanderplatz tonight”的两种评分机制之间的比较：

extracted Simple tags (score) BILOU tags (score)
near Alexanderplatz tonight loc loc time (3) B-loc L-loc U-time (3)
near Alexanderplatz tonight loc loc time (3) U-loc U-loc U-time (1)
near Alexanderplatz tonight O loc time (2) O U-loc U-time (1)
near Alexanderplatz tonight loc O time (2) U-loc O U-time (1)
near Alexanderplatz tonight loc loc loc (2) B-loc I-loc L-loc (1)

评估核心模型

您可以使用评估脚本在一组测试案例中评估训练有素的模型：

rasa test core --stories test_stories.md --out results

这会将失败的故事打印到results/failed_stories.md。如果至少有一项行动被错误地预测，我们会将任何故事视为失败。

此外，这会将混乱矩阵保存到名为的文件中 results/story_confmat.pdf。对于您域中的每个动作，混淆矩阵显示了正确预测该动作的频率以及相反地预测了一个不正确动作的频率。

比较核心的配置

要为您的核心模型选择配置，或为特定策略选择超参数，您想衡量Rasa Core将其推广 到以前从未见过的对话的程度。特别是在项目开始时，您没有太多真正的对话可用来训练您的机器人，因此您不只是想扔掉一些东西来用作测试集。

Rasa Core具有一些脚本来帮助您选择和调整策略配置。对它感到满意后，就可以在整个数据集上训练最终配置。为此，您首先必须针对不同的配置训练模型。创建两个（或更多）配置文件，包括要比较的策略，然后使用compare训练脚本的模式训练模型：

 rasa train core -c config_1.yml config_2.yml \
  -d domain.yml -s stories_folder --out comparison_models --runs 3 \
  --percentages 0 5 25 50 70 95

对于提供的每种策略配置，Rasa Core将接受多次培训，培训数据中不包括0、5、25、50、70和95％的培训案例。进行多次运行以确保结果一致。

该脚本完成后，您可以在compare 模式下使用评估脚本来评估您刚刚训练的模型：

rasa test core -m comparison_models --stories stories_folder
--out comparison_results --evaluate-model-directory

这将评估所提供故事中的每个模型（可以是训练集，也可以是测试集），并绘制一些图表以向您显示哪种策略效果最佳。通过评估整个故事，您可以衡量Rasa Core对预言故事的预测程度。

要比较单个策略，请创建每个仅包含一个策略的配置文件。如果您不确定要比较的策略，我们建议您试用 EmbeddingPolicy和KerasPolicy，看看哪种策略更适合您。

注意

这个训练过程可能会花费很长时间，因此我们建议让它在不中断的后台运行。

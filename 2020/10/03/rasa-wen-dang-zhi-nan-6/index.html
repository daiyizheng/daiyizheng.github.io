设置CI / CD

即使开发上下文助手与开发传统软件有所不同，您仍应遵循软件开发最佳实践。设置持续集成（CI）和持续部署（CD）管道可确保对您的漫游器进行增量更新可以改善它，而不会损害它。

概述

持续集成（CI）是一种经常合并代码更改并在提交更改时自动对其进行测试的做法。连续部署（CD）意味着将集成的更改自动部署到登台或生产环境。它们一起使您可以对助手进行更频繁的改进，并有效地测试和部署这些更改。

本指南将涵盖哪些应该在CI / CD管道，具体到一个项目拉莎。如何实现该管道取决于您。那里有许多CI / CD工具，例如GitHub Actions， GitLab CI / CD，Jenkins和 CircleCI。我们建议选择与您使用的任何Git存储库集成的工具。

持续集成（CI）

改善助手的最佳方法是频繁进行增量更新。无论变化有多小，您都希望确保它不会带来新的问题或对助手的性能产生负面影响。

通常最好对合并/提取请求或提交运行CI检查。大多数测试足够快，可以在每次更改时运行。但是，仅当某些文件已更改或存在某些其他指示符时，您才可以选择运行更多的资源密集型测试。例如，如果您的代码托管在Github上，则只有在拉取请求具有特定标签的情况下，您才能进行测试运行（例如“需要NLU测试”）。

验证数据和案例

数据验证可验证域文件，NLU数据或故事数据中是否没有错误或重大不一致。

rasa data validate --fail-on-warnings --max-history <max_history>

如果数据验证导致错误，则训练模型也将失败。通过包含该--fail-on-warnings标志，验证也将在不会出现问题的警告上失败，这些问题不会阻止训练模型，但可能指示混乱的数据，例如域中列出的未在任何故事中使用的操作。

数据验证包括故事结构验证。故事验证检查您是否有任何故事，其中相同的对话历史记录中会出现不同的机器人操作。故事之间的冲突将阻止模型学习正确的对话模式。将--max-history参数设置max_history为中的备注策略的值config.yml。如果尚未设置，请使用默认值5。

火车模型

rasa train

训练模型可验证您的NLU管道和策略配置有效且可训练，并且提供了用于测试对话的模型。如果它通过了CI测试，那么 作为持续部署过程的一部分，您还可以将经过训练的模型上传到服务器。

测试助手

在测试对话中测试训练有素的模型是对助手在某些情况下的行为有信心的最佳方法。这些以修改后的故事格式编写的故事，使您可以提供整个对话，并测试在给出此用户输入的情况下您的模型将以预期的方式运行。当您开始从用户对话中介绍更复杂的故事时，这一点尤其重要。

rasa test --stories tests/conversation_tests.md --fail-on-prediction-errors

该--fail-on-prediction-errors标志确保如果任何测试对话失败，则测试将失败。

端到端测试仅与您所包括的测试用例一样透彻和准确，因此，在对助手进行改进时，应继续增加测试对话集。遵循的一个很好的经验法则是，您应该以测试对话为目标，以代表真实对话的真实分布。Rasa X使得基于真实对话添加测试对话变得容易。

注：终端到终端的测试并没有执行你的操作代码。您将需要 在单独的步骤中测试操作代码。

比较NLU性能

如果您对NLU训练数据进行了重大更改（例如，将一个意图分为两个意图或添加了很多训练示例），则应运行 完整的NLU评估。您将需要比较没有更改的NLU模型的性能与更改。

您可以通过在交叉验证模式下运行NLU测试来做到这一点：

rasa test nlu --cross-validation

您还可以在训练集上训练模型并在测试集上进行测试。如果您使用训练测试集方法，则最好使用此CI步骤的一部分对数据进行改组和拆分，而不是使用静态NLU测试集，因为后者很容易过时。rasa data split

因为此测试不会导致通过/失败退出代码，所以最好使结果可见，以便您可以解释它们。例如，此工作流程 包括使用结果表注释PR，该结果表显示哪些意图与其他意图混淆。

由于NLU比较可能是相当耗费资源的测试，因此您可以选择仅在满足某些条件时才运行此测试。条件可能包括存在手动标签（例如，“需要进行NLU测试”），对NLU数据的更改或对NLU管道的更改。

测试操作代码

用于测试您的动作代码的方法将取决于其实现方式。例如，如果您连接到外部API，建议编写单元测试以确保这些API对常见输入的响应达到预期。无论您测试动作代码如何，都应将这些测试包括在CI管道中，以便每次进行更改时都可以运行。

持续部署（CD）

为了经常向用户提供改进，您将需要尽可能多的自动化部署过程。

一旦CI检查成功，CD步骤通常在推送或合并到某个分支上运行。

  • 部署您的Rasa模型
  • 部署动作服务器

部署您的Rasa模型

如果您在CI管道中运行端到端测试，那么您将已经拥有训练有素的模型。如果CI结果令人满意，您可以设置CD管道以将训练好的模型上传到Rasa服务器。例如，要将模型上传到Rasa X：

curl -k -F "model=@models/my_model.tar.gz" "https://example.rasa.com/api/projects/default/models?api_token=&#123;your_api_token&#125;"

如果您使用的是Rasa X，还可以将上载的模型标记 为active（如果使用多个部署环境，则可以标记为哪个部署）：

curl -X PUT "https://example.rasa.com/api/projects/default/models/my_model/tags/active"

但是，如果你的更新包括更改到这两个模型和你的操作代码，而这些变化以任何方式依赖于对方，你应该不会 自动标记模型production。首先，您将需要构建和部署更新的动作服务器，以便新模型不会（例如）调用更新前动作服务器中不存在的动作。

部署操作服务器

您可以自动 构建动作服务器的新映像并将其上传到映像存储库，以更新动作代码。如上所述，如果动作服务器与当前生产模型不兼容，则在将新的图像标签自动部署到生产时应格外小心。

CI / CD管道示例

作为示例，请参见Sara的CI / CD管道 ，您可以在Rasa Docs中与之交谈的Rasa助手以及 Carbon Bot。两者都使用Github Actions作为CI / CD工具。

这些示例只是众多可能性中的两个。如果您喜欢设置CI / CD，请在论坛上与Rasa社区共享。
